## 二、方案设计：自动快照回滚

### 2.1 快照机制（后端）

#### saveDatabase() 前置钩子

```
saveDatabase() 执行流程：

① createSnapshot()
   ├─ 检查快照目录是否存在，不存在则创建
   ├─ 生成时间戳文件名：2026-06-10T14-30-00.json
   ├─ 复制 db.json → db-snapshots/2026-06-10T14-30-00.json
   └─ 写入/更新快照索引 snapshots/index.json

② 原写入逻辑：fs.writeFileSync(dbPath, JSON.stringify(db))

③ cleanOldSnapshots()
   └─ 读取索引，保留最近 200 个，删除最旧的（上限可在 UI 中调整 50～1000）
```

#### 快照索引文件 (snapshots/index.json)

```json
[
  {
    "timestamp": "2026-06-10T14-30-00",
    "fileSize": 1048576,
    "dbTables": 15,
    "createdBy": "system"
  },
  {
    "timestamp": "2026-06-10T11-00-00",
    "fileSize": 1048500,
    "dbTables": 15,
    "createdBy": "system"
  }
]
```

#### 新增 IPC 通道

| 通道名 | 功能 | 参数 | 返回 |
|---|---|---|---|
| `db:snapshots:list` | 获取快照列表 | — | `{ success, data: SnapshotInfo[] }` |
| `db:snapshots:create` | 手动创建快照 | `{ label?: string }` | `{ success, data: SnapshotInfo }` |
| `db:snapshots:restore` | 还原到指定快照 | `{ timestamp: string }` | `{ success }` |
| `db:snapshots:delete` | 删除快照 | `{ timestamp: string }` | `{ success }` |

#### 还原流程

```
还原请求 → ① 锁定界面，阻止用户操作
         ② 备份当前 db.json → db-snapshots/pre-restore-YYYY-MM-DDTHH-mm-ss.json
         ③ 复制目标快照 → db.json
         ④ 重新加载数据库（db = JSON.parse(db.json)）
         ⑤ 解锁定界面
         ⑥ 触发前端重新加载数据
```

> 不需要重启应用！`dbReady = true` 后所有组件重新读取即可。

### 2.2 快照目录结构

```
<dataPath>/
├── db.json
├── db-snapshots/
│   ├── index.json
│   ├── 2026-06-10T14-30-00.json
│   ├── 2026-06-10T11-00-00.json
│   ├── pre-restore-2026-06-10T15-00-00.json   ← 还原前自动备份
│   └── ...
├── uploads/
└── ...
```

## 五、风险与注意事项

| 风险 | 缓解措施 |
|---|---|
| 快照文件占用磁盘 | 默认保留 200 个快照（约 200～400 MB），管理员可在 UI 中调整上限 |
| 还原时应用正在运行 | 不需要重启；还原后重新加载 db 对象 + 通知前端刷新 |
| 还原后上传文件存在但 db 无引用 | 文件不受影响，只是 db 中无记录；可手动清理 |
| 还原时断电导致 db.json 损坏 | 先写临时文件 `db.json.tmp`，成功后再 rename |
| 审计日志大量堆积 | 现有 MAX_LOGS = 10000，回滚时审计日志也随 db 回滚 |
