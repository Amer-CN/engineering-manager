## 📋 执行摘要

Phase 8 是工程管家的大版本升级（v2.12.0 → v3.0.0），聚焦**建筑劳务分包核心业务痛点**，通过三大方向（工资管理增强 + 项目指挥中心增强 + 数据安全与协作）提升系统实用性和数据安全性。

**核心驱动**：安岳县 2025 年高标准农田项目工资发放核对经验 + 建筑劳务分包行业通用痛点。

## 🔨 方向 A：工资管理增强（Priority ⭐⭐⭐⭐⭐）

### A1. 银行回单批量解析

**业务痛点**：目前只支持单张回单解析，安岳县项目有 100+ 工人，逐个上传效率极低。

**功能设计**：
1. **批量上传界面**
   - 支持拖拽/选择多个银行回单图片（jpg/png/pdf）
   - 实时显示解析进度（已解析 X/总数）
   - 解析失败自动重试（最多 3 次）

2. **智能匹配算法**
   - 解析出：交易日期、交易金额、交易对手（工人姓名）、交易流水号
   - 自动匹配：工人姓名 → `workers` 表（模糊匹配 + 手动确认）
   - 自动匹配：交易金额 → `wages` 表（金额 + 月份匹配）

3. **匹配结果确认界面**
   - 表格展示：回单信息 | 匹配工人 | 匹配工资记录 | 匹配置信度
   - 支持手动调整（下拉选择正确工人/工资记录）
   - 批量确认（一键确认所有高置信度匹配）

4. **数据回写**
   - 更新 `wages` 表的 `paymentStatus = 'paid'`
   - 更新 `wages` 表的 `paymentDate` 和 `paymentReceipt`
   - 写入 `auditLogs` 审计日志

**技术实现**：
- 前端：新增 `BankReceiptBatch.tsx` 组件（批量上传 + 匹配确认）
- 后端：扩展 `wages:parseBankReceipt` IPC 通道（支持批量）
- OCR：复用现有百度 OCR / Tesseract.js 离线识别
- 数据：SQLite 双写（已有基础设施）

**预期收益**：
- 100 工人工资发放核对：从 2 小时 → 10 分钟
- 降低人工错误率（从 ~5% → <1%）

### A2. 工资条自动生成

**业务痛点**：工人需要签字确认工资条，目前需要手动制作 Excel/Word，效率低且易出错。

**功能设计**：
1. **工资条模板管理**
   - 内置标准模板（符合建筑行业规范）
   - 支持自定义模板（上传 Word/HTML 模板）
   - 模板变量：`{{姓名}}`、`{{月份}}`、`{{基础工资}}`、`{{加班费}}`、`{{扣款}}`、`{{实发}}` 等

2. **批量生成工资条**
   - 选择项目 + 月份 → 自动生成所有工人工资条
   - 支持按班组筛选
   - 实时显示生成进度

3. **工资条预览与导出**
   - 在线预览（HTML 渲染，样式与打印一致）
   - 导出 PDF（每个工人单独 PDF + 批量 ZIP 打包）
   - 导出 Excel（汇总表，方便财务存档）

4. **工资条确认流程**（v3.0 范围）
   - 导出 PDF 后打印，工人手动签字
   - 确认状态可手动标记（`wages.confirmed = true`）
   - **手机扫码签名功能 defer 到 v3.1**（涉及本地 HTTP 服务或云端部署，复杂度较高）

**技术实现**：
- 前端：新增 `WageSlipGenerator.tsx` 组件（模板管理 + 批量生成 + 预览）
- 后端：新增 `wages:generateSlips` IPC 通道（查询数据 + 渲染 HTML + 导出 PDF）
- PDF：**使用 Electron 内置 `webContents.printToPDF()`**（无需额外依赖，Electron 已内置 Chromium）
- 模板：复用现有 `mammoth` 逻辑（Word → HTML）

**预期收益**：
- 100 工人工资条制作：从 3 小时 → 5 分钟
- 符合劳动监察要求（工资条存档）

### A3. 工资发放记录

**业务痛点**：目前工资发放记录分散（银行回单 + 工资表 + 审计日志），缺乏统一的发放流水视图。

**功能设计**：
1. **工资发放流水表**
   - 字段：发放日期、发放方式（银行转账/现金）、发放金额、工人姓名、项目名称、银行回单号、确认状态
   - 数据来源：`wages` 表（paymentStatus = 'paid'）+ `auditLogs`（操作记录）

2. **发放记录查询**
   - 按项目筛选、按月份筛选、按工人筛选
   - 支持模糊搜索（工人姓名/银行回单号）
   - 显示统计：总发放金额、已确认金额、未确认金额

3. **发放记录导出**
   - 导出 Excel（符合财务存档要求）
   - 导出 PDF（打印版，带公司公章位置）

4. **发放异常预警**
   - 同一工人同一月份多次发放 → 标黄警告
   - 发放金额与工资记录不匹配 → 标红警告
   - 未确认工资条超过 7 天 → 标橙提醒

**技术实现**：
- 前端：新增 `WagePaymentRecords.tsx` 组件（表格 + 查询 + 导出）
- 后端：新增 `wages:getPaymentRecords` IPC 通道（JOIN 查询）
- 数据：SQLite 查询优化（已有 `wageQueries.getWageStats()` 可扩展）

**预期收益**：
- 财务对账效率提升 50%
- 降低发放错误风险

### A4. 欠薪预警

**业务痛点**：建筑行业欠薪是高风险问题，目前缺乏系统性的预警机制。

**功能设计**：
1. **欠薪规则配置**
   - 规则 1：工资记录生成超过 X 天未发放 → 预警
   - 规则 2：工人离职超过 X 天未结清工资 → 预警
   - 规则 3：项目资金余额不足以支付下月工资 → 预警
   - 规则可自定义（Settings 页面配置）

2. **欠薪预警仪表盘**
   - 显示当前欠薪工人列表（姓名、欠薪金额、欠薪天数、风险等级）
   - 按项目分组、按风险等级排序
   - 支持导出欠薪清单（Excel/PDF）

3. **欠薪预警通知**
   - 应用内通知（顶部通知栏）
   - 邮件通知（可选，需要配置 SMTP）
   - 微信/钉钉通知（可选，需要配置 Webhook）

4. **欠薪处理跟踪**
   - 标记处理状态（待处理/处理中/已解决）
   - 记录处理备注（谁处理的、如何处理、处理结果）
   - 处理完成后自动关闭预警

**技术实现**：
- 前端：新增 `WageArrearsWarning.tsx` 组件（仪表盘 + 配置 + 通知）
- 后端：新增 `wages:getArrearsWarnings` IPC 通道（计算欠薪逻辑）
- 通知：扩展现有 `Toast` 系统 + 新增 `NotificationSettings.tsx`
- 数据：SQLite 查询（JOIN `wages` + `workers` + `projects`）

**预期收益**：
- 降低劳动监察风险
- 提升工人满意度（及时发放工资）

## 📊 方向 B：项目指挥中心增强（Priority ⭐⭐⭐⭐）

### B1. 成本预警系统

**业务痛点**：目前成本超预算只能事后发现，缺乏实时预警。

**功能设计**：
1. **预算配置**
   - 每个项目可配置总成本预算 + 分类预算（人工/材料/机械/其他）
   - 预算数据存储在 `projects` 表的 `budget` 字段（JSON）
   - **⚠️ 待确认**：`projects` 表是否已存在 `budget` 字段？如不存在需先执行数据库迁移

2. **成本实时计算**
   - 从 `costLedger` 表实时聚合实际成本
   - 分类成本：根据 `category` 字段自动分类
   - 计算偏差率：（实际成本 - 预算成本）/ 预算成本 × 100%

3. **预警规则**
   - 偏差率 > 80% → 黄色预警（提醒）
   - 偏差率 > 100% → 红色预警（禁止继续支出）
   - 预警显示在项目指挥中心和仪表盘

4. **预警通知**
   - 应用内通知（顶部通知栏）
   - 邮件通知（可选）
   - 预警记录写入 `auditLogs`

**技术实现**：
- 前端：修改 `ProjectCommandCenter.tsx`（新增成本预警模块）
- 后端：扩展 `statsQueries.getDashboard()`（含预算偏差计算）
- 数据：SQLite 查询优化（实时聚合）

**预期收益**：
- 降低成本超支风险
- 提升项目管理精细化水平

### B2. 利润预测

**业务痛点**：目前利润只能事后计算，缺乏前瞻性预测能力。

**功能设计**：
1. **利润预测模型**
   - 输入：已发生成本 + 剩余工作量（%） + 合同总金额
   - 计算：预测总成本 = 已发生成本 + （已发生成本 / 完成进度%）× (1 - 完成进度%)
   - 计算：预测利润 = 合同总金额 - 预测总成本

2. **利润预测界面**
   - 显示：合同金额、已发生成本、完成进度%、预测总成本、预测利润
   - 可视化：利润趋势图（折线图，显示预测利润变化）
   - 支持手动调整参数（剩余工作量%、预计追加成本）

3. **利润预警**
   - 预测利润 < 0 → 红色预警（亏损风险）
   - 预测利润率 < 5% → 黄色预警（利润过低）
   - 预警显示在项目指挥中心

**技术实现**：
- 前端：修改 `ProjectCommandCenter.tsx`（新增利润预测模块）
- 后端：新增 `projects:getProfitForecast` IPC 通道（计算逻辑）
- 可视化：复用现有 `recharts`（折线图/柱状图）

**预期收益**：
- 提升项目决策能力（及时调整成本策略）
- 降低亏损风险

### B3. 材料消耗分析

**业务痛点**：目前材料管理只有入库/出库记录，缺乏消耗分析和库存预警。

**功能设计**：
1. **材料消耗统计**
   - 按材料类型统计消耗量（入库 - 出库 = 库存）
   - 按项目统计材料成本占比
   - 显示消耗趋势图（折线图，按月份）

2. **库存预警**
   - 库存 < 安全库存 → 黄色预警（需要补货）
   - 库存 = 0 → 红色预警（停工风险）
   - 预警规则可配置（Settings 页面）

3. **材料消耗报表**
   - 导出 Excel（材料消耗明细表）
   - 导出 PDF（材料消耗汇总表）
   - 支持按项目/按材料类型/按月份筛选

**技术实现**：
- 前端：新增 `MaterialConsumption.tsx` 组件（统计 + 预警 + 报表）
- 后端：新增 `materials:getConsumptionStats` IPC 通道（聚合查询）
- 数据：SQLite 查询（JOIN `materials` + `projects`）

**预期收益**：
- 降低材料浪费（精准采购）
- 降低停工风险（及时补货）

### B4. 劳务人员画像

**业务痛点**：目前工人信息只有基础档案，缺乏技能/出勤率/工资水平等维度的分析。

**功能设计**：
1. **工人技能画像**
   - 字段：技能类型（钢筋工/木工/电工等）、技能等级（初级/中级/高级）、证书编号

2. **工人出勤率统计**
   - 计算出勤率：实际出勤天数 / 应出勤天数 × 100%

   - 出勤率 < 80% → 标黄警告（可能影响工期）

3. **工人工资水平分析**
   - 按技能类型统计平均工资
   - 按项目统计工人工资分布
   - 显示工资趋势图（折线图，按月份）

4. **工人推荐派工**
   - 根据技能匹配度 + 出勤率 + 工资水平推荐工人
   - 推荐结果显示在派工界面

**技术实现**：
- 前端：修改 `WorkerManagement.tsx`（新增人员画像 Tab）
- 后端：扩展 `workerQueries.getWorkerStats()`（含技能/出勤率/工资统计）
- 数据：SQLite 查询（JOIN `workers` + `attendances` + `wages`）

**预期收益**：
- 提升派工效率（精准匹配技能）
- 降低用工风险（避免出勤率低的工人）

### C2. 数据自动备份

**业务痛点**：目前只有手动快照，缺乏定期自动备份机制。

**功能设计**：
1. **自动备份配置**
   - 备份频率：每天/每周/每月（可配置）
   - 备份时间：指定时间点（如每天凌晨 2:00）
   - 备份保留数量：最多保留 X 个备份（默认 7 个，超过自动删除最旧的）

2. **备份存储位置**
   - 默认：应用数据目录下的 `backups/` 文件夹
   - 可选：外部存储（U 盘/移动硬盘/网络驱动器）

3. **备份恢复**
   - 显示所有备份（日期/大小/备注）
   - 支持一键恢复（恢复前自动创建当前状态快照）

**技术实现**：
- 前端：修改 `Settings.tsx`（新增自动备份配置界面）
- 后端：扩展 `database.ts`（含定时备份逻辑，使用 `node-schedule`）
- **⚠️ 注意事项**：Electron 打包后应用可能休眠，建议配合持久化机制（应用启动时检查是否有遗漏的备份任务）
- 备份：复用现有快照逻辑（`create-snapshot.js`）

**预期收益**：
- 降低数据丢失风险
- 符合建筑行业数据合规要求

### C3. 多用户权限细化

3. **权限校验**

**技术实现**：
- 前端：修改 `Settings.tsx`（用户管理模块，新增角色字段）
- 后端：修改 `ipc-guard.ts`（含角色权限映射）
- 数据：SQLite 修改（`users` 表新增 `role` 字段）

**预期收益**：
- 提升数据安全性（细粒度权限控制）

### C4. 数据导出加密

**业务痛点**：目前导出的 Excel/PDF 没有密码保护，存在数据泄露风险。

**功能设计**：
1. **导出加密配置**
   - Settings 页面新增"导出加密"开关
   - 配置导出密码（默认与登录密码不同）
   - 可选：每个导出文件使用不同密码（随机生成）

2. **Excel 导出加密**
   - 使用 `exceljs` 库（开源，原生支持密码加密）
   - 加密后的 Excel 需要输入密码才能打开

3. **PDF 导出加密**
   - 使用 Electron 内置 `webContents.printToPDF()` 的加密选项
   - 加密后的 PDF 需要输入密码才能打开

4. **加密导出流程**
   - 用户点击导出 → 弹出密码输入框（可选）
   - 使用默认密码或用户输入的密码加密
   - 加密完成后提示"导出成功（已加密）"

**技术实现**：
- 前端：修改所有导出功能（Excel/PDF）的代码
- 后端：扩展导出逻辑（含加密算法）

## 🔧 技术选型（修订版）

| 技术点 | 选型 | 理由 |
|--------|------|------|
| **PDF 生成** | Electron `webContents.printToPDF()` | Electron 已内置 Chromium，无需额外下载（puppeteer 需下载 ~170MB Chromium） |
| **Excel 加密** | `exceljs` | `xlsx` 免费版不支持加密（需付费 Pro 版），`exceljs` 开源且原生支持密码加密 |
| **定时任务** | `node-schedule` | 轻量级，适合 Electron 主进程；需配合启动时检查遗漏任务 |
| **可视化图表** | `recharts`（已有） | 复用现有依赖，保持技术栈统一 |
| **PDF 预览** | `react-pdf-viewer` | 如需在线预览 PDF 时使用 |

## 🚨 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **批量 OCR 识别准确率下降** | 匹配错误率高 | 提供手动调整界面 + 置信度阈值 |
| **PDF 生成性能问题** | 100 人工资条生成耗时过长 | 使用队列 + 分批生成 + 进度提示 |
| **定时备份失败** | 数据丢失风险 | 备份前检查磁盘空间 + 备份失败通知用户 |
| **权限细化导致用户体验下降** | 操作变复杂 | 提供角色模板（一键配置） |
| **`projects.budget` 字段不存在** | B1 功能无法实施 | 需在 Phase 8.3 之前先执行数据库迁移 |
| **扫码签名功能复杂度超预期** | 延迟整体进度 | 已 defer 到 v3.1，v3.0 只做手动确认 |
| **`exceljs` 与现有 `xlsx` API 不兼容** | 需要改造现有导出代码 | 先在小范围试点，确认兼容性后再全量替换 |
